## Important decisions
- do not store secrets directly in `~/.pi`
- do not rely on `.bashrc` for the Zen key
- use the `!cat /run/secrets/opencode_zen_api_key` pattern in `agent/models.json`
- treat Pi itself as Nix-managed through `~/nixos-config`, not npm-managed
- keep Pi packages declarative in `agent/settings.json` even though the actual package payload is fetched at runtime
- use `CONTEXT7_API_KEY` from NixOS-managed exported env vars rather than storing Context7 credentials in dotfiles
- use `pi-mcp-adapter` for MCP support and keep the richer built-in file tools enabled via an extension
- prefer tracked Pi config plus ignored runtime caches/session/auth state in dotfiles

## Cross-machine caveats
- the current `npmCommand` uses a hardcoded writable prefix at `/home/aj/.pi/agent/npm`
- this works well as long as the username/path remains `aj` on other machines
- if the username or home path differs on a future machine, Pi package installation should be switched to a wrapper script or another path abstraction rather than keeping the hardcoded path
- Context7 requires `CONTEXT7_API_KEY` to be exported in the shell; after `nixos-rebuild switch`, existing shells may need:

      unset __NIXOS_SET_ENVIRONMENT_DONE
      . /etc/set-environment

- that refresh can still print permission errors for unrelated secrets such as `hf_token` while correctly refreshing readable vars like `CONTEXT7_API_KEY` and `OPENCODE_ZEN_API_KEY`
